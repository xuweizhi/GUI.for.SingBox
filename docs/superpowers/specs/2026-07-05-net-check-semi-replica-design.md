# 网络检测半复刻设计

## 背景

`/data/karing` 的“网络检测”不是单一探测，而是一组分阶段诊断能力的集合。它在一个页面里串行执行：

- IPv4 / IPv6 联通性检测；
- 当前代理出站可用性检测；
- 多类 DNS 检测；
- 目标域名 DNS 查询；
- 规则集状态检查；
- HTTP 访问检测；
- 路由表展示。

GUI.for.SingBox 当前已经落地的是“精简版网络检测”，覆盖：

- 核心状态；
- 代理 HTTP 连通性；
- TCP 连通性；
- 节点测速面板。

这一版已经解决了最基础的“核心是否正常、代理是否可用、目标端口是否可达”问题，但和 Karing 相比仍缺 3 类排障信息：

- 目标域名 DNS 解析结果；
- 本次检测命中的规则与链路；
- 当前配置引用规则集的可用状态。

当前项目已有能力足以支撑一个“半复刻”版本：

- 现有 `NetCheckView` 页面骨架；
- 现有 `useRuntimeNetworkCheck` 和 `useNetworkCheck` 检测流程；
- Clash API `/dns/query`、`/rules`、`/providers/rules`、`/connections`、`/proxies/{proxy}/delay`；
- 现有 `useNodeController`、`kernelApiStore.proxies`；
- 本地 `rulesetsStore` 持有的规则集元数据。

因此，本期目标不是 1:1 搬运 Karing 页面，而是在不新增重型后端专用接口的前提下，把 GUI.for.SingBox 现有网络检测升级成“信息结构接近 Karing、实现复杂度可控”的增强版。

## 已确认需求

- 继续使用现有 `NetCheck` 页面入口，不新增第二个诊断页。
- 桌面端和 Headless WebUI 使用同一套页面和同一套诊断逻辑。
- 保留当前精简版的三项检查，不回退现有能力。
- 补充 Karing 中最有价值、且当前项目已有能力足以支撑的诊断信息。
- 不为了这次增强版新增复杂 Go 服务层或额外长链路 API。
- 页面结果必须适合“快速判断故障落在哪一层”，而不是只堆接口响应。

## 目标

- 把当前网络检测从“3 个基础探测”升级成“4 组分层诊断”。
- 让用户能在一次检测里同时看到：
  - 核心和代理链是否工作；
  - 目标域名解析到了什么；
  - 当前主出站的大致健康状况；
  - 本次请求命中了哪条规则、走了哪条链；
  - 当前配置引用的规则集是否可用。
- 在不补重型后端接口的前提下，尽可能逼近 Karing 页面解决问题的效率。

## 非目标

- 不完整复刻 Karing 的所有子项。
- 不实现 `resolver / direct / proxy` 三类 DNS 延迟分项测试。
- 不实现系统路由表展示。
- 不新增“默认路由 DNS 查询”这类专用 bridge 接口。
- 不新增 sing-box 外部 service 或自定义 RPC 网关。
- 不把网络检测做成首页弹窗或设置页子标签。

## 方案对比

### 方案 1：只在现有结果区增加更多平铺结果

继续沿用现有单列表模型，把新项直接加成更多 `result item`。

优点：

- 改动最小；
- 不需要改页面结构。

缺点：

- 当结果项增加到 8 个以上时，可读性明显下降；
- DNS、规则、规则集这几类信息的层级关系会混在一起；
- 不利于以后继续扩展。

### 方案 2：保留单页，结果区改成分组卡片

继续使用 `NetCheckView`，但把结果区拆成固定分组，每组内部是多条结果。

优点：

- 维持当前入口与页面路由；
- 分组后更接近 Karing 的“分阶段诊断”阅读习惯；
- 后续补项时不需要再改页面结构。

缺点：

- 需要调整当前 `useNetworkCheck` 的结果模型；
- 要额外设计组级与项级状态。

### 方案 3：做成“概览 + 明细折叠面板”

默认只展示每组一条摘要，点开再看细节。

优点：

- 首屏更紧凑。

缺点：

- 当前页面数据量还没有大到必须折叠；
- 会增加交互状态和测试成本；
- 不利于 WebUI 一眼复制和截图排障。

## 推荐方案

采用方案 2：保留单页，结果区改成分组卡片。

这是当前最平衡的选择。它不会改变已上线入口，也不要求后端先大幅演进；同时它把新增信息组织成更稳定的阅读结构，后续再补更深的 DNS / 路由诊断也不会返工页面骨架。

## 核心设计

### 1. 页面结构

页面仍由 3 个大区域组成：

1. 顶部输入区
   - 目标地址输入框；
   - `开始检测` 按钮；
   - `清空结果` 按钮；
   - 检测中禁用态。
2. 结果区
   - 改为 4 组卡片；
   - 每组卡片展示组标题、组摘要、组内结果条目。
3. 节点诊断区
   - 保留现有 `NodeDiagnosticsPanel`；
   - 继续承担单节点测速和整组测速能力。

### 2. 结果分组

结果区固定为 4 组，顺序不可配置：

1. `概览`
2. `DNS`
3. `出站与规则`
4. `规则集状态`

这样用户的阅读路径固定为：

1. 先看核心和链路是否通；
2. 再看域名解析是否正常；
3. 再看当前实际命中了哪条规则；
4. 最后看配置依赖的规则集是否存在明显异常。

### 3. 概览组

概览组保留当前精简版的三项：

- `核心状态`
- `代理 HTTP`
- `TCP 连通`

实现沿用现有逻辑：

- 核心状态：`probeApiAvailability()`；
- 代理 HTTP：`Requests` / `HttpHead` / `HttpGet` 通过内核代理；
- TCP 连通：`TcpPing(host:port)`。

这组仍然是全页的前置判断，不调整语义。

### 4. DNS 组

DNS 组新增一条目标域名查询结果，数据来源是当前 controller 的 `/dns/query`。

展示内容：

- 查询目标域名；
- 查询状态；
- 解析结果中的 A / AAAA 记录；
- 响应里的 `Server` 字段；
- 响应里的 `Status`。

行为规则：

- 当目标输入最终解析为域名时执行；
- 当目标输入是纯 IPv4 / IPv6 地址时，该组标记为 `skipped`，摘要明确写“目标为 IP，跳过 DNS 查询”；
- 不额外做本地系统 `dns.lookup` 对照；
- 不推断 `default-router` 或 DNS 命中规则。

### 5. 出站与规则组

这一组拆成两条结果：

#### 5.1 当前主出站延迟

展示内容：

- 当前主选择组名称；
- 当前链路摘要；
- 最终叶子节点名称；
- 当前可用延迟。

数据来源：

- 复用 `useNodeController` 的 `primary`；
- 如果 `primary.leaf` 上已有有效 `history`，直接读取最后一条延迟；
- 如果没有有效延迟，则在检测流程中对 `primary.leafName` 额外调用一次 `getProxyDelay()`。

行为规则：

- `direct` 模式下仍展示当前模式和链路，但延迟条目标记为 `skipped`；
- 如果叶子节点不存在或 controller 数据不完整，标记为 `failed` 并给出明确说明；
- 不在这一步执行整组测速，避免和现有节点面板职责重叠。

#### 5.2 最近命中的规则与链路

展示内容：

- 本次目标请求命中的 `rule`；
- 匹配到的 `chains`；
- 匹配连接的主机与端口。

数据来源：

- 在代理 HTTP 检测执行结束后，调用 `getConnections()`；
- 从返回的连接列表中匹配“最可能属于本次检测”的最近连接。

匹配策略固定为：

1. 优先匹配 `metadata.host === target.host`；
2. 其次匹配 `destinationIP === target.host`（当目标本身是 IP 时）；
3. 端口需要等于目标端口；
4. 在满足以上条件的候选连接中，取 `start` 最新的一条。

行为规则：

- 如果代理 HTTP 这一步根本没执行成功，则该条结果标记为 `skipped`；
- 如果代理 HTTP 成功但连接列表中没找到可匹配记录，则标记为 `failed`，摘要写“未找到可匹配连接记录”；
- 这条结果是基于连接列表推断，不宣称“绝对等价于 Karing 的 outboundQuery”。

### 6. 规则集状态组

这一组用来回答“当前 profile 依赖的规则集本地状态是否正常”。

数据来源：

- `profilesStore.currentProfile.route.rule_set`
- `rulesetsStore.rulesets`

展示逻辑：

- 遍历当前 profile 引用的 `route.rule_set`；
- 只关注 `type === 'local'` 或 `type === 'remote'` 的条目；
- 用 `path` 去 `rulesetsStore.rulesets` 里查本地规则集实体；
- 对每个引用输出一条结果。

每条结果展示：

- 规则集名称或 tag；
- 类型（local / remote）；
- 本地是否找到对应实体；
- 是否 disabled；
- `count`；
- `updateTime`。

状态判定：

- 能找到实体且未禁用：`success`
- 找到实体但已禁用：`failed`
- 当前 profile 引用了规则集但本地实体缺失：`failed`
- 当前 profile 未引用任何本地 / 远程规则集：整组输出一条 `skipped`

明确取舍：

- 不主动触发规则集更新；
- 不查询远端下载过程；
- 不把“更新时间很久以前”直接定义为失败，只展示时间信息。

## 目标地址解析规则

沿用现有单输入框模型，接受：

- `https://example.com/path`
- `http://example.com`
- `example.com`
- `example.com:8443`
- `1.1.1.1`
- `[2606:4700:4700::1111]:443`

统一解析规则：

1. 输入包含 `http://` 或 `https://` 时按 URL 解析。
2. 不含 scheme 时默认按 HTTPS 目标处理。
3. 若显式指定端口，则 TCP 与连接匹配都使用该端口。
4. 纯 IP 输入允许通过，但 DNS 组自动跳过。
5. 解析失败时阻断整轮检测。

## 数据模型

### 1. 组模型

结果区新增组级模型，至少包含：

- `id`
- `title`
- `status`
- `summary`
- `items`

其中：

- `status` 是组级汇总状态；
- `items` 是组内条目列表。

### 2. 项模型

组内条目沿用当前结果项思路，至少包含：

- `id`
- `title`
- `status`
- `summary`
- `detail`
- `durationMs`

### 3. 组级状态汇总规则

固定规则：

- 存在 `failed` 条目时，组状态为 `failed`
- 否则存在 `running` 条目时，组状态为 `running`
- 否则存在 `success` 条目时，组状态为 `success`
- 否则全部为 `skipped` 时，组状态为 `skipped`

这保证组状态始终可由条目状态确定，不引入额外隐式逻辑。

## 运行模型

一次 `开始检测` 会创建一轮新的检测会话，按固定顺序执行：

1. 解析输入
2. 执行概览组
3. 执行 DNS 组
4. 执行出站与规则组
5. 执行规则集状态组
6. 刷新节点诊断区

选择串行而不是并行，原因是：

- 规则匹配依赖代理 HTTP 已经触发过真实连接；
- 组间顺序稳定，便于用户阅读；
- 失败原因更容易和前置条件对应起来。

## 交互设计

### 1. 输入区

- 默认值保留稳定 HTTPS 目标；
- 输入框 placeholder 明确支持 URL、域名、IP 与 `host:port`；
- 检测进行中禁用输入和按钮；
- 清空仅清结果，不重置输入。

### 2. 分组卡片

每组卡片展示：

- 标题；
- 组级状态；
- 一行组摘要；
- 组内条目列表。

每条结果展示：

- 状态图标；
- 名称；
- 摘要；
- 可选详情；
- 可选耗时。

不引入折叠面板，首版全部直出。

### 3. 节点诊断区

保留当前行为：

- 单节点测速；
- 整组测速；
- 节点切换；
- 只读模式提示。

本次增强不改变其职责，只在页面检测完成后额外触发一次刷新。

## 错误处理

- 输入解析失败：阻断整轮检测。
- 核心状态失败：DNS、规则匹配、当前主出站延迟、规则集状态都可继续执行各自可执行部分，但依赖 controller 的项需要明确标记失败原因。
- 代理 HTTP 未执行成功：最近命中的规则与链路标记 `skipped`。
- 目标是 IP：DNS 组标记 `skipped`。
- 当前 profile 不引用规则集：规则集状态组标记 `skipped`。
- 单项异常只影响该项，不中断整轮检测。

## 测试策略

至少覆盖以下层级：

### 1. 纯函数测试

覆盖：

- URL / 域名 / IP / `host:port` / IPv6 输入；
- 端口推断；
- 非法输入阻断。

### 2. composable 测试

覆盖：

- 概览三项成功；
- DNS 域名查询成功；
- 目标为 IP 时 DNS 跳过；
- 代理 HTTP 成功但连接列表无匹配记录；
- 当前主出站无历史延迟时回退到主动测速；
- 当前 profile 未引用规则集；
- 引用规则集但本地实体缺失。

### 3. 视图测试

覆盖：

- 4 组卡片渲染顺序；
- 组摘要与条目状态展示；
- 检测中禁用态；
- 清空结果；
- 节点面板在核心不可用时的提示不回归。

## 风险与取舍

### 1. 连接匹配不是精确原生路由查询

当前项目没有 Karing 的专用 `outboundQuery` 能力，这版通过 `/connections` 推断最近匹配连接。

取舍：

- 对排障已经足够有用；
- 必须在文案与实现里承认它是“最近匹配连接”，不是绝对权威路由查询。

### 2. DNS 只有最终查询结果，没有 Karing 式分流解释

当前 controller 的 `/dns/query` 可以给出解析结果，但不会直接告诉用户命中了哪类 DNS 路径。

取舍：

- 首版优先把“解析成功 / 返回了哪些地址”展示出来；
- 暂不伪造 DNS 规则解释。

### 3. 规则集状态基于本地元数据而不是远端下载实时状态

这会比 Karing 的“下载中 / 下载失败”语义弱一些。

取舍：

- 不主动触发远端更新；
- 先展示本地实体是否存在、是否禁用、最后更新时间和数量。

## 结论

这版“网络检测半复刻”继续使用现有 `NetCheckView`，把结果区升级成 4 组分层诊断：

- 概览；
- DNS；
- 出站与规则；
- 规则集状态。

它保留当前已验证通过的核心 / 代理 HTTP / TCP 检查，同时基于现有 Clash API、节点控制器和规则集 store 补上 Karing 中最有价值的 3 类信息。这样可以在不新增重型后端接口的前提下，把 GUI.for.SingBox 的网络检测提升到“足够接近 Karing、又保持实现边界清晰”的水平。
