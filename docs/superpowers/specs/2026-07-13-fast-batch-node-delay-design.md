# 批量节点测速提速设计

## 背景

首页节点控制器目前让单节点测速和批量测速共用连续重试策略。每个节点最多请求 3 次，单次超时默认是 5 秒，相邻失败尝试间隔 300 ms。

该策略能提高偶发网络故障下的成功率，但也会放大批量测速的完成时间。不可用节点的最坏耗时约为 `5 秒 × 3 + 300 ms × 2 = 15.6 秒`。即使批量请求受并发池控制，失败节点仍可能让整批测速等待多个超时周期。

## 目标

- 批量测速时，每个节点只请求 1 次。
- 手动单节点测速继续保留最多 3 次尝试。
- 不改变批量测速的并发上限、取消语义、进度统计和刷新流程。
- 错误状态记录实际尝试次数，避免把批量单次失败显示为 `3/3`。

## 非目标

- 不调整用户配置的测速 URL、超时时间或并发数量。
- 不修改 sing-box Clash API 的请求格式。
- 不为请求增加 `AbortSignal`，也不强制中断已经发出的请求。
- 不修改现有完整控制器 `GroupsController` 的测速行为。
- 不改变错误分类规则和不可测速节点过滤规则。

## 方案

为 `useNodeController` 内部的 `runNodeTest` 增加最大尝试次数选项。该选项只影响一次节点测速操作，不写入设置或持久化配置。

- 默认最大尝试次数为 `MAX_DELAY_ATTEMPTS`，即 3 次。
- `testNode` 使用默认值，保持手动单节点测速行为不变。
- `testGroup` 调用时传入 1 次，使批量测速不重试。

该方案复用现有探测流程，只改变调用上下文提供的尝试上限，不复制单次测速逻辑。

## 数据流

### 手动单节点测速

1. `testNode` 检查节点是否正在测速。
2. `runNodeTest` 最多调用延迟 API 3 次。
3. 任意一次返回正延迟后立即成功。
4. 三次均失败时保存最后一条错误，错误状态中的 `attempts` 为 3。
5. 测速结束后刷新代理快照。

### 批量测速

1. `testGroup` 过滤不可测速节点和正在测速的节点。
2. 现有异步池按用户配置的并发上限调度节点。
3. 每个池任务调用 `runNodeTest`，最大尝试次数为 1。
4. 单次请求成功或失败后立即完成该节点，并更新批量统计。
5. 全部任务结束后统一刷新代理快照。

## 错误与状态

失败结果继续使用现有错误标准化及分类逻辑。`NodeDelayError.attempts` 必须记录本次操作实际执行的请求次数：

- 手动单节点测速连续失败：`attempts` 为 3。
- 批量测速单次失败：`attempts` 为 1。
- 在后续尝试中成功：成功状态继续记录实际成功尝试次数。

界面不得固定假设失败总尝试次数为 3。批量测速失败应展示 `1/1`，手动测速连续失败应展示 `3/3`。为此，节点错误状态同时保存已执行次数和本次最大尝试次数，或提供语义等价的数据，使组件能展示真实分子与分母。

## 取消语义

现有异步池取消逻辑保持不变：

- 取消后不再启动排队节点。
- 已发出的 API 请求允许自然结束。
- 已取消且未完成的节点不计入成功或失败。

由于批量测速不再重试，活动节点完成当前请求后不会进入重试等待，取消响应也会更直接。

## 测试

控制器回归测试覆盖：

- 批量测速中的失败节点只调用延迟 API 1 次。
- 多个批量节点各自只请求 1 次，成功和失败统计仍按节点计算。
- 批量失败错误记录实际尝试次数 1。
- 手动单节点失败仍请求 3 次，并记录 3 次尝试。
- 手动单节点首次失败后仍可重试成功。
- 批量取消仍阻止排队节点启动。

组件测试覆盖：

- 批量单次失败显示 `1/1`。
- 手动连续失败显示 `3/3`。
- 成功、未测试及错误提示展示保持不变。

## 兼容性与风险

- 不修改持久化设置，因此现有用户配置无需迁移。
- 批量测速对瞬时网络抖动的容错率会下降，这是优先缩短完成时间的明确权衡。
- 手动单节点测速保留重试，可用于复核批量测速中的失败节点。
- 改动集中在首页节点控制器和对应测试，不影响完整控制器及其他网络诊断功能。
