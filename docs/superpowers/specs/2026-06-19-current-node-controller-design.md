# 首页当前节点控制设计

## 背景

GUI.for.SingBox 已通过 Sing-Box Clash API 提供完整的运行时节点控制能力：

- `GET /proxies` 读取选择组、当前选择、嵌套选择链和历史延迟。
- `PUT /proxies/{group}` 切换 `Selector` 当前节点。
- `GET /proxies/{proxy}/delay` 测试单个节点延迟。
- 现有 `GroupsController` 支持选择组展开、节点切换、单节点测速和整组测速。
- 桌面端与 Headless WebUI 均已具备访问核心控制端口的 HTTP 通道。

当前主要问题不是能力缺失，而是入口不明显：首页只展示流量、连接数、内存和路由模式，用户必须进入页面底部的“控制器”才能看到当前选择并切换节点。Karing 则把当前节点作为首页核心状态，并从该状态直接进入节点选择。

## 已确认需求

- 首页直接显示当前连接节点。
- 点击首页卡片后打开精简节点选择弹窗。
- 支持手动切换节点。
- 支持单节点测速和整组测速。
- 默认展示“主选择组”，同时允许在弹窗中切换到其他选择组。
- 不替换或重构现有完整 `GroupsController`。
- 复用现有 Clash API 和前端 store，不新增 Go 后端接口。

## 概念边界

“当前节点”指 Sing-Box 选择器的当前控制状态，即从主选择组沿 `now` 字段解析得到的选择链和最终叶子节点。

它不表示所有并发连接必然经过同一个节点。规则模式下，不同路由规则可以使用不同出站；已经建立的连接也可能仍保留旧链路。每条连接的实际出站链继续以现有连接列表中的 `chains` 字段为准。

## 目标

- 让用户无需进入完整控制器即可识别当前主节点。
- 将节点切换和测速缩短为首页卡片到弹窗的一步操作。
- 在桌面端和 Headless WebUI 中提供一致体验。
- 保留现有高级控制器，降低改动范围和行为回归风险。

## 非目标

- 不移除或重写现有 `GroupsController`。
- 不修改订阅、配置生成或 Sing-Box 出站结构。
- 不实现基于实时连接流量推断“使用最多节点”的统计。
- 不把不同路由规则使用的所有实际出站合并成一个虚假的全局节点。
- 不为 `URLTest` 组增加手动选择能力。
- 不新增节点收藏、最近使用或节点详情编辑。

## 方案对比

### 方案 1：首页卡片 + 精简节点弹窗

首页新增当前节点卡片，点击后打开专用弹窗。弹窗复用现有 API，但采用更紧凑的节点选择交互。

优点：

- 最接近 Karing 的首页体验。
- 常用操作路径最短。
- 不影响现有完整控制器。
- 可以独立控制移动端布局和加载状态。

缺点：

- 节点列表逻辑会与现有控制器存在少量展示层重复。
- 需要抽取共享的主组和选择链解析逻辑。

### 方案 2：首页卡片直接打开现有控制器

卡片只作为现有 `GroupsController` 的新入口。

优点：

- 改动最小。
- 不新增节点列表组件。

缺点：

- 完整控制器信息密度较高，不符合首页快速切换场景。
- 用户仍需展开选择组，体验与 Karing 差距明显。

### 方案 3：重构完整控制器为共享组件

先拆分现有 `GroupsController`，再由首页弹窗和完整控制器共同组合底层组件。

优点：

- 长期重复最少。
- 两个入口可以完全共享节点卡片和测速状态。

缺点：

- 本次改动范围和回归风险最高。
- 会把一个可独立交付的首页功能扩大为控制器重构。

## 推荐方案

采用方案 1：首页卡片 + 精简节点弹窗。

共享部分只抽取无 UI 的选择组解析和节点操作逻辑，现有 `GroupsController` 保持不动。这样可以在低风险下完成首页体验，同时为未来逐步共享组件保留边界。

## 架构设计

### 1. 组件划分

新增两个前端单元：

- `CurrentNodeCard`：负责首页状态展示和打开弹窗。
- `NodeSelectorModal`：负责选择组切换、节点搜索、排序、节点切换和测速。

新增一个无 UI 的 composable 或工具模块，例如 `useNodeController`，职责包括：

- 从 `kernelApiStore.proxies` 生成可展示选择组。
- 解析主选择组。
- 安全解析选择链和最终节点。
- 统一刷新 `/proxies`。
- 封装节点切换、单节点测速和整组测速状态。

`OverView` 只负责布置卡片并承载弹窗，不直接实现节点控制细节。

### 2. 数据来源

运行时状态以 `kernelApiStore.proxies` 为唯一事实来源。配置结构只用于判断默认路由出口和隐藏组：

- `kernelApiStore.config.mode`：当前 `rule`、`global` 或 `direct` 模式。
- `profilesStore.currentProfile.route.final`：规则模式的默认出站 ID。
- `profilesStore.currentProfile.outbounds`：将出站 ID 映射为运行时 tag，并读取 `hidden`、`icon` 等展示属性。
- `kernelApiStore.proxies`：读取组类型、`all`、`now`、节点类型和延迟历史。

不直接从订阅列表推断运行时节点，因为订阅数据不包含核心当前选择状态。

### 3. 主选择组解析

主组解析必须是确定性的：

1. `global` 模式优先使用运行时 `GLOBAL` 组。
2. `direct` 模式返回只读的 `direct` 状态，不提供切换。
3. `rule` 模式把当前 profile 的 `route.final` 从出站 ID 映射为 tag，以该 tag 作为解析起点。
4. 从起点沿运行时 `now` 链向下查找，返回遇到的第一个 `Selector`。
5. 如果默认路由无法映射、对应运行时出站不存在或链中没有 `Selector`，回退到第一个未隐藏的 `Selector`。
6. 如果仍无结果，卡片显示“未检测到可切换节点”，弹窗仍可展示可发现的其他组。

回退排序沿用 `/proxies` 返回顺序，不额外按名称重排，避免每次刷新后主组无故变化。

### 4. 选择链解析

从目标组开始重复读取当前项的 `now`，直到遇到没有 `now` 的叶子节点。结果包含：

- 组名。
- 选择链中的每一级名称。
- 最终叶子节点。
- 最终节点最近一次有效延迟。

解析器必须维护已访问名称集合，并设置不大于代理总数的最大深度。发现循环引用或缺失节点时终止解析，返回已解析部分和异常状态，禁止无限循环。

示例：

```text
代理 -> 自动选择 -> 香港 02
```

其中“代理”为主 `Selector`，“自动选择”为 `URLTest`，“香港 02”为最终叶子节点。

### 5. 刷新策略

- 核心启动或首页激活时立即刷新一次 `/proxies`。
- 首页处于激活状态且核心运行时，每 5 秒刷新一次 `/proxies`。
- 页面失活、组件卸载或核心停止时终止定时刷新。
- 打开弹窗时立即刷新。
- 节点切换成功后立即刷新。
- 单节点或整组测速完成后立即刷新。
- 同一时间只允许一个常规刷新请求在途，避免定时刷新和用户操作重复请求。

测速过程中使用本地状态立即显示结果，不等待下一次轮询。

## 交互设计

### 1. 首页卡片

在现有实时流量、总流量、连接数和内存卡片区域加入“当前节点”卡片。卡片具有可点击样式，并包含：

- 标题：“当前节点”。
- 主选择组名称。
- 选择链或最终节点名称。
- 最终节点最近一次延迟。
- 加载、不可用或解析异常状态。

如果空间不足，选择链单行省略，完整内容通过 tooltip 提供。卡片不能只用颜色表达延迟状态，必须保留数值或文本。

状态示例：

- `代理 / 香港 02 / 86 ms`
- `代理 / 自动选择 -> 香港 02 / 86 ms`
- `直连 / direct`
- `未检测到可切换节点`
- `核心控制接口不可用`

### 2. 节点选择弹窗

弹窗顶部包含：

- 选择组下拉框。
- 搜索输入框。
- 按延迟排序开关。
- “整组测速”按钮。
- 测速进度和取消入口。

弹窗默认选中主选择组。选择组列表包含未隐藏的 `Selector` 和 `URLTest`，并始终把运行时存在的 `GLOBAL` 作为系统组选项。全局模式下 `GLOBAL` 排在首位，其他模式下排在普通组之后。

### 3. 节点列表

每个节点卡片展示：

- 节点名称。
- 节点类型。
- UDP 能力（存在时）。
- 最近延迟或“未测速”“超时/不可用”。
- 当前选择标记。

交互规则：

- `Selector` 成员可点击切换。
- 当前已选节点不可重复提交切换请求。
- 点击延迟区域只测试该节点，不触发切换。
- `URLTest` 组展示自动选中的成员和组内节点，但节点卡片不可手动切换。
- `direct` 模式下弹窗以只读状态打开，允许查看和测速，但禁用所有节点切换，并提示当前流量固定使用直连。
- 切换期间禁用当前组的其他切换操作，测速可继续显示但不重复触发同一节点。
- 搜索按节点名称做不区分大小写匹配。
- 延迟排序按有效延迟升序排列，无结果和失败项置于末尾，并保持原始相对顺序。

### 4. 响应式布局

- 窄屏使用单列列表。
- 中等宽度使用两列。
- 宽屏根据弹窗可用宽度使用三至四列。
- 顶部工具栏在窄屏自动换行，不产生横向滚动。
- 弹窗内容区独立滚动，标题和工具栏保持可见。

### 5. 可访问性和国际化

- 卡片、组选择、测速和当前节点状态提供明确的可访问名称。
- 键盘可以聚焦节点卡片，并使用 Enter 或 Space 执行切换。
- 当前选中状态不能只依赖背景色。
- 新文案同时加入中文和英文 locale。
- 动态状态使用现有消息组件，不新增仅鼠标可见的反馈。

## 节点切换行为

节点切换复用现有 `handleUseProxy` 行为：

1. 仅允许 `Selector` 切换。
2. 调用 `PUT /proxies/{group}`，body 为 `{ name: proxy }`。
3. 如果已启用“自动关闭连接”，关闭 `chains` 中包含该选择组的现有连接。
4. 切换成功后刷新 `/proxies`。
5. 切换失败时保留旧状态并显示具体错误。

弹窗不得先乐观修改 `group.now`，避免 API 失败时短暂显示错误选择。

## 测速行为

### 1. 单节点测速

- 调用现有 `GET /proxies/{proxy}/delay`。
- 使用应用设置中的测试 URL 和超时时间。
- 请求进行时只标记对应节点。
- 成功后显示返回的毫秒值。
- 失败时记录独立失败状态，显示“超时/不可用”，不得显示 `0 ms`。

### 2. 整组测速

- 对当前组 `all` 中的节点执行测速。
- 复用现有并发上限设置。
- 显示总数、完成数、成功数和失败数。
- 允许用户取消尚未开始的任务。
- 已完成结果保留；取消不回滚已完成结果。
- 同一节点在一次批量任务中只测试一次。

### 3. URLTest

`URLTest` 的自动选择由 Sing-Box 管理。本功能只读取其 `now` 和测速结果，不通过 `PUT /proxies/{group}` 手动指定成员。

## 错误处理

- 核心未运行：卡片显示不可用状态，点击后给出“请先启动核心”提示。
- Clash API 不可达：保留最后一次成功状态并标记为已过期，弹窗提供重试。
- 鉴权失败：沿用现有 WebUI 未授权处理。
- 主组解析失败：显示回退状态，不阻止查看其他组。
- 切换失败：保留旧选择，解除 loading，并显示 API 错误。
- 测速失败：只影响对应节点，不中断其他节点测速。
- 批量测速取消：停止调度未开始任务，已在途请求自然结束。
- 组件卸载：停止轮询并取消本地任务状态更新，避免卸载后写入。

## 对现有代码的影响

预计涉及：

- `frontend/src/views/HomeView/components/OverView.vue`：加入当前节点卡片。
- 新增首页当前节点卡片组件。
- 新增精简节点选择弹窗组件。
- 新增主组和选择链解析模块或 composable。
- `frontend/src/lang/locale/zh.ts` 和 `en.ts`：增加文案。
- 新增解析逻辑和组件测试。

保持不变：

- Go Bridge 和 Headless 核心代理接口。
- `frontend/src/api/kernel.ts` 的现有 API 协议。
- 现有 `GroupsController`。
- 配置生成、订阅和出站编辑流程。

如果实现中需要导出当前未导出的通用类型或常量，可以做最小范围调整，但不得借机重构无关模块。

## 测试策略

### 1. 单元测试

主组解析覆盖：

- `global` 模式选择 `GLOBAL`。
- `direct` 模式返回只读 `direct`。
- `rule` 模式从 `route.final` ID 映射到运行时 tag。
- 默认链为 `Selector -> URLTest -> 节点`。
- 默认链包含多个 `Selector`。
- 默认出站缺失后的回退。
- 隐藏组不参与回退。
- 缺失 `now` 目标。
- 循环引用和最大深度保护。

排序和过滤覆盖：

- 名称搜索不区分大小写。
- 有效延迟升序。
- 未测速和失败节点置后。
- 相同延迟保持稳定顺序。

### 2. 组件测试

- 首页加载后展示主组、选择链和最终节点。
- 核心未运行时显示不可用状态。
- 点击卡片打开弹窗并刷新代理数据。
- 弹窗默认选中主组并可切换其他组。
- `direct` 模式下弹窗允许测速但禁止节点切换。
- `Selector` 节点切换成功后刷新状态。
- 切换失败时保留旧节点并显示错误。
- `URLTest` 节点不可手动切换。
- 单节点测速成功和失败状态。
- 整组测速的进度、并发限制和取消。
- 搜索、排序和空结果。
- 定时刷新在卸载后停止。

### 3. 回归验证

运行：

```bash
cd frontend
pnpm test
pnpm type-check
pnpm build
```

手动验证桌面端和 Headless WebUI：

- 规则、全局和直连三种模式。
- 普通 `Selector` 和嵌套 `URLTest`。
- 切换节点后首页、弹窗、完整控制器和托盘状态一致。
- 开启“自动关闭连接”时，切换后受影响连接被关闭。
- 手机宽度下弹窗无横向溢出。

## 实施顺序建议

1. 实现并测试主组与选择链解析模块。
2. 实现节点控制 composable 和刷新生命周期。
3. 实现精简节点选择弹窗。
4. 实现首页当前节点卡片并接入 `OverView`。
5. 补齐中英文文案和可访问性。
6. 执行自动化测试、构建和桌面/WebUI 手动回归。

## 验收标准

- 核心运行时，首页无需进入完整控制器即可看到主选择组和最终节点。
- 规则、全局和直连模式下卡片状态符合主组解析规则。
- 点击卡片可以查看选择组和节点列表。
- 用户可以在 `Selector` 中手动切换节点，失败时不会显示错误的已选状态。
- 用户可以测试单节点和当前整组延迟，并区分成功、未测速和失败。
- `URLTest` 自动选择状态可见但不可被手动覆盖。
- 首页卡片、精简弹窗、完整控制器和托盘读取同一运行时选择状态。
- 桌面端和 Headless WebUI 均可使用，移动端宽度下布局可操作。
- 现有完整控制器、连接列表和核心启动流程无行为回归。
