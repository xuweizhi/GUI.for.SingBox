# WebUI Token 登录页设计

## 背景

当前项目的 Headless WebUI 只有 `token` 鉴权能力，没有账号密码登录、登录页、退出入口或统一的未授权回退流程。后端会在鉴权通过后下发 `HttpOnly Cookie` `gfs_webui_token`，并允许通过查询参数 `token`、Cookie 或 `Authorization: Bearer <token>` 完成校验。

现有前端直接进入应用初始化流程，未登录状态下也会立刻触发 RPC、SSE 和核心接口访问，不适合直接承载一个需要先认证、再初始化的浏览器端登录体验。

## 已确认需求

- 登录方式：保留现有 `token` 机制，不新增账号密码体系。
- 交互形式：新增登录页，用户手动输入 `token`。
- 持久化：用户一次性输入后长期记住。
- 失效处理：当 `token` 错误或服务端 `token` 已变更时，自动跳回登录页，并提示「token 无效，请重新输入」。
- 用户操作：提供明确的「退出登录 / 更换 token」入口。

## 目标

- 为 WebUI 提供独立登录页。
- 在不修改后端鉴权模型的前提下，为浏览器端补齐可用的登录体验。
- 统一处理未授权状态，避免应用在 401 后停留在半初始化或错误状态。
- 尽量复用现有后端 `token` 和 Cookie 逻辑，避免引入新的认证协议。

## 非目标

- 不引入账号密码、用户表、JWT 或 Session。
- 不改变桌面端启动路径和桌面端 Bridge 逻辑。
- 不在本次范围内实现多用户管理、权限分级或 `token` 生成 UI。

## 方案对比

### 方案 1：独立 `/login` 路由 + 本地持久化 + 启动前校验

用户进入 WebUI 时，先通过路由守卫或启动门禁判断是否已完成 `token` 校验。未通过时跳转到 `/login`；登录页提交 `token` 后，前端先向服务端发起一次同源校验请求，让后端写入 `HttpOnly Cookie`，成功后再进入现有应用。

优点：

- 结构清晰，和现有 `vue-router` 模式一致。
- 登录态、重定向、主动退出都容易表达。
- 可以把应用初始化明确放在鉴权成功之后，减少错误状态。

缺点：

- 需要补一个轻量认证状态层。
- 要调整 `App.vue` 当前「加载即初始化」的执行顺序。

### 方案 2：顶层登录遮罩，不新增路由

应用始终挂载，只是在最上层覆盖一个登录层，未通过认证前禁止使用主界面。

优点：

- 少改 router。

缺点：

- 与当前 `useAppBootstrap()` 启动时机冲突较大。
- 未授权情况下仍可能过早触发数据加载。
- 退出登录和回退状态比路由方案更难收口。

### 方案 3：URL 参数引导登录

首次要求用户通过带 `token` 的 URL 访问，前端读取并保存，再跳回普通 URL。

优点：

- 首次登录输入更少。

缺点：

- `token` 会进入地址栏、历史记录或分享链路，安全性较差。
- 不适合作为默认主路径。

## 推荐方案

采用方案 1：独立 `/login` 路由 + `localStorage` 持久化 + 启动前校验。

原因：

- 最符合当前前端结构。
- 最容易实现「长期记住」「401 自动回登录页」「主动退出登录」。
- 可以最大化复用后端现有 `token` 与 Cookie 机制。

## 设计细节

### 1. 路由设计

新增一个公开路由：

- `/login`：WebUI 登录页。

现有业务路由保持不变，但都视为受保护路由。访问受保护路由时：

- 若认证状态已确认有效，正常进入。
- 若本地无保存的 `token`，跳转 `/login`。
- 若本地有保存的 `token` 但尚未校验，先执行校验，成功后放行，失败后跳转 `/login`。

为了兼容用户在未登录时直接打开任意页面，应保留目标路由信息。登录成功后优先返回原始目标页，否则回到 `/`。

### 2. 认证状态模型

新增一个仅用于 WebUI 的前端认证 store，例如 `webuiAuth`，负责以下职责：

- 读取本地持久化的 `token`。
- 保存新的 `token`。
- 清空本地 `token`。
- 执行远程校验。
- 记录认证状态，例如 `idle`、`checking`、`authenticated`。
- 记录最近一次未授权原因，供登录页提示。

本地持久化使用 `localStorage`，键名应使用明确前缀，例如 `gfs.webui.token`，避免与其他设置混淆。

### 3. 校验协议

不新增后端登录接口，直接复用现有鉴权中间件。

前端校验方式：

- 向同源地址发起一个轻量请求，例如 `HEAD /?token=<用户输入的 token>`。
- 请求命中后端鉴权逻辑。
- 若 `token` 正确，后端返回成功状态，并写入 `HttpOnly Cookie` `gfs_webui_token`。
- 若 `token` 错误，后端返回 `401 Unauthorized`。

选择这种方式的原因：

- 现有后端已经支持通过查询参数传入 `token`。
- 浏览器的 `EventSource` 无法稳定附带自定义鉴权头，但可以自动携带同源 Cookie。
- 先完成一次校验并落下 Cookie，后续 RPC、SSE 与核心接口就可以直接走同源请求。

### 4. 应用启动门禁

当前 `App.vue` 在组件初始化阶段就调用 `useAppBootstrap()`。本次需要改为「认证成功后再启动」。

建议结构：

- WebUI 模式下：
  - 先初始化认证状态。
  - 若未认证，仅渲染路由视图中的登录页，不执行应用数据加载。
  - 若认证成功，再执行一次应用 bootstrap。
- 桌面模式下：
  - 保持现有启动行为不变。

这样可以避免未认证时立刻触发以下行为：

- `GetEnv()`
- 应用设置读取
- 配置、订阅、规则集、插件等初始化
- 运行时事件流建立

### 5. 登录页交互

登录页至少包含：

- `token` 输入框
- 「登录」按钮
- 错误提示区域

交互规则：

- 输入为空时不提交。
- 提交后按钮进入 loading 状态，避免重复点击。
- 校验成功后：
  - 保存 `token` 到 `localStorage`
  - 跳转原始目标页或首页
- 校验失败后：
  - 不保存 `token`
  - 显示「token 无效，请重新输入」

文案层面不引入「账号」「密码」等术语，避免与真实账号体系混淆。

### 6. 未授权回退

需要统一处理 401。

处理规则：

- 清空本地持久化的 `token`
- 重置认证状态
- 跳转 `/login`
- 显示提示「token 无效，请重新输入」

统一拦截点应覆盖 WebUI 场景下的主要入口：

- 浏览器端 Bridge RPC 封装
- 浏览器端事件发送封装
- 浏览器端核心 API 请求封装

SSE 基于 `EventSource`，失败时拿不到和 `fetch` 一样完整的错误结构，因此不要求 SSE 自己独立弹错。只要后续任一 RPC 或核心请求命中 401，就必须完成统一回退。

### 7. 退出登录 / 更换 token

在 WebUI 主界面提供一个明确入口，点击后执行：

- 清空本地持久化的 `token`
- 重置前端认证状态
- 跳转 `/login`

服务端 `HttpOnly Cookie` 无法由前端直接删除。本次不新增后端登出接口，原因如下：

- 后端当前使用的是单一静态 `token` 校验，而非服务端会话。
- 前端清空本地 `token` 后，用户已无法继续完成新的登录校验流程。
- 当服务端 `token` 更新后，旧 Cookie 在后续请求中也会因校验失败而失效。

如果后续需要严格的「主动清除 Cookie」语义，再单独增加后端注销接口。

### 8. 对现有代码结构的影响

预计需要调整的区域：

- 路由定义：新增 `/login` 与受保护路由策略。
- `App.vue`：把 bootstrap 时机改为受认证状态控制。
- WebUI 浏览器端 Bridge：补齐统一未授权处理。
- 新增认证 store：管理 `token`、状态与回退逻辑。
- 新增登录页组件。
- 主界面中的退出登录入口。

桌面模式相关文件保持原样，不引入行为回归。

### 9. 安全性与权衡

本方案把 `token` 保存在 `localStorage`，不是高强度安全方案，但符合当前需求「一次性输入后长期记住」，且复杂度最低。

已知权衡：

- `localStorage` 中保存的是明文 `token`。
- 浏览器同源环境若发生 XSS，理论上可以读取该值。

本次接受该权衡，理由：

- 当前项目本身已依赖单一 `token` 模型。
- 需求优先级是可用性与低改动面。
- 如果未来要提升安全性，应转向短期票据或服务端会话，而不是在前端对静态 `token` 做伪加密。

### 10. 测试策略

应覆盖以下场景：

- 未保存 `token` 时访问首页，自动跳转 `/login`。
- 输入正确 `token`，成功进入目标页。
- 输入错误 `token`，停留在登录页并显示错误提示。
- 保存过 `token` 后刷新页面，仍可直接进入应用。
- 服务端 `token` 变更后，下一次受保护请求触发 401，自动回登录页并提示。
- 点击「退出登录 / 更换 token」后，立即回到登录页。
- 桌面模式启动流程不受影响。

### 11. 实施顺序建议

1. 新增认证 store 和 `localStorage` 读写逻辑。
2. 新增 `/login` 路由与登录页。
3. 改造 `App.vue` 的启动门禁。
4. 在 WebUI 请求入口补齐 401 回退。
5. 增加主界面的退出登录入口。
6. 做 WebUI 与桌面模式回归验证。

## 验收标准

- WebUI 首次访问时能看到登录页。
- 用户输入正确 `token` 后可以进入应用，并在浏览器重开后继续保持可访问。
- 当 `token` 无效或服务端 `token` 已变更时，会自动返回登录页，并显示「token 无效，请重新输入」。
- 用户可以主动退出登录并重新输入新的 `token`。
- 桌面端现有行为保持不变。
