# 统一节点测速调度设计

## 背景

项目目前有两套节点测速实现：

- 首页节点弹窗通过 `useNodeController.ts` 调度单节点和批量测速。
- 完整控制器通过 `GroupsController.vue` 独立调度单节点和策略组测速。

两处都逐节点调用 Clash API 的 `/proxies/{name}/delay` 接口，但在重试、错误分类、取消、进度统计和结果刷新方面行为不一致。最近的优化已将首页弹窗的批量测速改为每个节点只请求 1 次，缩短了不可用节点造成的等待时间，但也降低了瞬时网络故障下的成功率；完整控制器仍保留另一套独立逻辑。

Karing 同样逐节点调用 Clash API，但使用全局队列控制测速任务。PC 端默认并发数量为 20，任一请求结束后立即调度下一个节点。可借鉴的重点是统一调度和持续补位，而不是更换测速接口。

## 目标

- 首页节点弹窗和完整控制器共用统一的测速服务。
- 默认维持 20 并发，并继续支持用户配置并发数量。
- 首轮覆盖所有节点，失败补测不得阻塞尚未开始的首轮节点。
- 只对超时、连接重置、EOF 和临时网络不可达等瞬时错误补测 1 次。
- DNS、认证、TLS/Reality、连接拒绝等确定性错误不补测。
- 同一节点被多个入口同时提交时，共享底层请求，不重复测速。
- 常规网络下，100 个节点的首轮测速以 10 秒内完成为性能目标，包括补测在内总体以约 15 秒内完成为目标。
- 两个入口统一延迟结果、错误分类、尝试次数、进度和取消语义。

## 非目标

- 不修改 sing-box 或 Clash API 的请求格式。
- 不新增策略组批量测速内核接口。
- 不修改网络诊断页面的探测流程。
- 不为批量任务设置强制 10 秒或 15 秒截止时间。
- 不强制中断已经发出的 HTTP 请求。
- 不增加出口 IP 解析等与延迟测速无关的 Karing 功能。
- 不进行与测速统一无关的组件重构。

## 方案选择

### 采用方案：统一测速服务与两阶段调度

新增与 Vue 组件无关的共享节点测速服务。首页节点弹窗和完整控制器只负责选择节点、提交任务、呈现进度及应用结果，不再分别实现请求调度和重试判断。

共享服务采用两阶段队列：

1. 首测队列包含本次任务中所有待测节点。
2. 瞬时失败节点进入补测队列，最多补测 1 次。
3. 调度器始终优先消费首测队列；首测队列为空后才消费补测队列。
4. 任一活动请求结束后立即补充新的请求，直到达到并发上限或队列清空。

该方案兼顾速度和成功率，并从根本上消除两套测速逻辑继续分叉的风险。

### 未采用方案：只共享并发池

仅改造 `createAsyncPool` 虽然改动较小，但两个入口仍会分别维护错误分类、取消、结果刷新和进度状态，无法解决行为不一致的根本问题。

### 未采用方案：使用策略组批量测速接口

把整组测速交给内核可以减少前端请求数量，但难以实现逐节点进度、选择性补测和任务级取消，并引入内核版本兼容风险。Karing 当前也采用逐节点请求和应用层队列调度，因此本次不采用该方向。

## 组件职责

### 共享测速服务

共享测速服务负责：

- 维护全局活动请求和排队请求。
- 根据用户配置限制全局并发数量，默认值为 20。
- 在请求结束后持续补位，不等待固定批次整体完成。
- 优先调度首测，低优先级调度补测。
- 判断错误是否允许补测。
- 对相同测速参数的同一节点执行请求去重。
- 向每个订阅任务分发节点状态和最终结果。
- 隔离不同任务的取消和进度。

共享服务不直接访问 Vue Store，不直接修改 `proxy.history`，也不负责界面文案。

### 首页节点控制器

`useNodeController.ts` 负责：

- 过滤不可测速节点和当前不可用的操作。
- 向共享服务提交单节点或批量任务。
- 将共享结果映射到现有本地延迟、尝试次数和错误状态。
- 维护节点弹窗的批量进度。
- 在任务结束后执行 1 次代理快照刷新。

现有无差别最多 3 次重试改为共享服务定义的选择性补测：确定性错误首轮即结束，瞬时错误最多执行 2 次。

### 完整控制器

`GroupsController.vue` 负责：

- 向共享服务提交单节点或策略组任务。
- 使用共享事件维护节点和策略组的加载状态。
- 使用共享最终结果更新展示。
- 在任务结束后执行 1 次代理快照刷新。

完整控制器不再直接调用 `getProxyDelay`，也不再自行维护并发池及成功、失败统计。

## 任务与请求模型

### 任务

每次单节点或批量操作创建独立任务。任务至少包含：

- 唯一任务标识。
- 去重后的节点列表。
- 测速 URL 和单次请求超时。
- 提交时的并发配置。
- 节点状态回调。
- 任务完成回调。
- 取消状态。

任务独立统计总数、最终完成数、成功数、失败数和等待补测数。

### 请求去重

请求去重键由节点名称、测速 URL 和单次请求超时共同组成。只有去重键完全一致的活动请求才能共享，避免不同测试参数的操作错误复用结果。

当多个任务订阅同一活动请求时：

- 底层只调用 1 次 `getProxyDelay`。
- 请求结果分别推进每个订阅任务的节点状态。
- 某个任务取消时只移除该任务的订阅，不影响其他任务。
- 没有任何任务继续订阅时，已发出的请求仍允许自然结束，但结果不写入已取消任务。

## 调度流程

### 批量测速

1. 调用方过滤不可测速节点并去重。
2. 共享服务为所有节点创建首测项。
3. 调度器按并发上限持续启动首测请求。
4. 请求成功后立即发布最终成功结果。
5. 请求出现确定性错误后立即发布最终失败结果。
6. 请求出现瞬时错误后，将节点放入补测队列并发布“待补测”状态。
7. 只要仍有未开始的首测节点，调度器就不启动补测节点。
8. 首测队列清空后，调度器使用空闲并发槽执行补测。
9. 补测成功时发布最终成功结果；补测失败时使用最后一次错误发布最终失败结果。
10. 所有节点获得最终结果或任务被取消后，任务结束。
11. 调用方统一执行 1 次代理快照刷新。

补测前不增加固定的 300 ms 等待。首测优先队列和请求排队本身提供短暂间隔，避免固定等待进一步拉长整批耗时。

### 单节点测速

单节点测速使用相同服务和错误策略：

- 首次成功立即结束。
- 确定性错误立即失败。
- 瞬时错误补测 1 次。
- 第二次仍失败时返回最后一次错误。

单节点操作与批量操作共享同一全局并发和请求去重机制，不建立特殊的高并发通道。

## 错误分类与补测策略

沿用现有错误标准化和分类逻辑，增加“是否可补测”的明确判断。

| 错误分类 | 是否补测 | 说明 |
| --- | --- | --- |
| `timeout` | 是 | 可能由短时拥塞或探测站响应抖动导致 |
| `connection-reset` | 是 | 包含连接重置和 EOF |
| `network-unreachable` | 是 | 仅对现有分类中的临时网络不可达补测 |
| `dns` | 否 | DNS 配置或域名解析失败通常不会在紧邻请求中恢复 |
| `authentication` | 否 | 凭据或协议认证错误不应重复请求 |
| `tls` | 否 | TLS、证书或 Reality 错误不应重复请求 |
| `connection-refused` | 否 | 目标明确拒绝连接时不应重复请求 |
| `unknown` | 否 | 缺少证据时不推测为瞬时错误 |

错误判断只依据 API 返回的信息。通用错误不自动补测，避免把确定性故障放大为额外请求。

## 状态与界面

每个任务内的节点状态包括：

- `queued`：等待首测。
- `testing`：请求执行中。
- `retry-queued`：首次瞬时失败，等待补测。
- `success`：获得有效正延迟。
- `failed`：获得最终错误。
- `cancelled`：任务取消且该节点尚未产生最终结果。

界面统一展示：

- 首次成功：`82 ms`。
- 补测成功：`82 ms · 2 次`。
- 首次确定性失败：错误分类及 `1/1`。
- 补测后最终失败：错误分类及 `2/2`。
- 等待首测：排队中。
- 等待补测：待补测。
- 活动请求：测速中。

批量进度中的 `completed` 只统计获得最终成功或最终失败的节点。进入 `retry-queued` 的节点尚未完成。任务取消后，尚未产生最终结果的节点不计入成功或失败，并保留测速前的界面结果。

## 结果一致性

- 测速结果首先通过本地状态实时展示，不等待代理快照刷新。
- 同一共享请求完成时，所有有效订阅者收到一致结果。
- 两个入口使用同一结果结构，不再分别解释错误或尝试次数。
- 轮询刷新不得清除仍在测试、等待补测或刚完成但尚未执行任务尾部刷新的本地结果。
- 每个批量任务结束后最多触发 1 次代理快照刷新。
- 共享服务不直接写入 `proxy.history`；调用方通过统一适配逻辑应用结果，避免重复写入。

## 取消语义

- 取消任务后，不再启动只属于该任务的排队首测或补测请求。
- 已发出的请求允许自然结束。
- 已取消任务不消费后续请求结果，也不写入新的失败状态。
- 如果活动请求仍有其他任务订阅，请求和其他任务正常继续。
- 取消一个入口的任务不影响另一个入口的任务或手动单节点测速。
- 任务取消前已经获得的最终结果可以保留；未完成节点恢复或保留测速前状态。

## 性能目标

性能目标以 100 个节点、默认 20 并发和常规可用网络为基准：

- 首轮测速争取在 10 秒内完成。
- 包含瞬时错误补测的总体耗时争取控制在约 15 秒内。
- 请求结束后立即补位，不因固定批次中单个慢请求阻塞其他并发槽。
- 补测不得抢占尚未开始的首测节点。

以上是运行目标，不是强制墙钟截止时间。实际耗时受用户配置的单次超时、测速 URL、节点质量和网络状况影响。服务不会在 10 秒或 15 秒时强制把活动请求判为失败。

## 测试策略

### 共享服务单元测试

采用测试驱动开发，先编写失败测试，再实现共享服务。覆盖：

- 并发数量从不超过配置上限。
- 任一请求结束后立即补位，不等待其他活动请求结束。
- 首测节点始终优先于补测节点。
- 超时、连接重置、EOF 和网络不可达进入补测。
- DNS、认证、TLS、连接拒绝和未知错误不补测。
- 每个瞬时失败节点最多补测 1 次。
- 相同去重键的并发任务只产生 1 个底层 API 请求。
- 测速 URL 或超时不同的请求不共享。
- 一个任务取消后不影响共享请求的其他订阅任务。
- 取消后不启动该任务尚未排队执行的节点。
- 成功、失败、待补测和取消状态的统计正确。

### 入口集成测试

首页节点控制器和完整控制器分别覆盖：

- 两个入口消费相同的成功和失败结果结构。
- 补测成功显示实际尝试次数 2。
- 最终失败显示正确分类及 `1/1` 或 `2/2`。
- 批量结束只刷新 1 次代理快照。
- 轮询刷新不覆盖受保护的本地结果。
- 不可测速节点仍被过滤。
- 单节点与批量并发时不产生重复请求。
- 取消操作不写入未完成节点的失败结果。

### 性能测试

使用虚拟计时器和可控请求 Promise 验证 100 个节点在 20 并发下持续补位，并验证补测不会阻塞首测。CI 不使用真实网络墙钟断言“10 秒内完成”，避免测速站和运行环境波动导致不稳定测试。

真实网络性能作为手工验收项，记录节点数量、并发设置、测速 URL、首轮完成时间、总完成时间、首次成功数和补测成功数。

## 兼容性与迁移

- 沿用现有测速 URL、超时时间和并发数量设置，不增加持久化字段。
- 默认并发数量仍为 20。
- 沿用现有 Clash API，因此不要求升级内核。
- 现有错误分类和本地化文案尽量复用，只增加队列状态及必要文案。
- `createAsyncPool` 可继续服务其他功能；本次不强制删除或全局重构该工具。

## 风险与缓解

- **全局去重状态泄漏：** 请求完成后必须在统一的 `finally` 路径移除活动请求，并由测试覆盖成功、失败和取消分支。
- **任务取消相互影响：** 活动请求与任务订阅分离，取消只移除订阅，不直接取消共享请求。
- **并发配置在任务中途变化：** 每个任务保存提交时配置，但全局调度器使用当前有效上限；实现计划需要明确并发上限更新策略并保持不超过用户最新配置。
- **本地结果被轮询覆盖：** 两个入口共享受保护节点集合或等价机制，直到任务尾部刷新结束。
- **补测增加总请求量：** 只补测有证据表明可能瞬时恢复的错误，并限制为 1 次。
- **两个入口迁移期间行为短暂分叉：** 实现按共享服务、首页入口、完整控制器的顺序推进，每一步都有回归测试，最终移除入口内独立调度逻辑。

## 验收标准

- 首页节点弹窗和完整控制器均通过共享服务测速。
- 默认并发不超过 20，且用户配置仍生效。
- 100 个常规节点的首轮测速以 10 秒内完成为目标，总体以约 15 秒内完成为目标。
- 瞬时错误最多补测 1 次，确定性错误不补测。
- 补测不阻塞任何尚未开始的首测节点。
- 两个入口同时测试相同节点和参数时只发出 1 个底层请求。
- 取消一个任务不会取消或污染其他任务。
- 两个入口的延迟、错误分类、尝试次数和进度语义一致。
- 自动化测试覆盖调度、去重、补测、取消、状态和入口集成行为。
